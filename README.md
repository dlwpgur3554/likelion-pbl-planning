## 2. 문제 정의

### 2.1 현황

알고리즘 트레이딩 시장은 빠르게 성장하고 있지만, **개인 투자자는 이 기술의 혜택을 받지 못하고 있다**. 기존 자동매매 플랫폼은 자산을 플랫폼에 예치해야 하고, 특정 거래소에만 종속되어 있어 보안 리스크와 불편함이 크다.

### 2.2 원인 분석

| 원인 | 영향 |
|---|---|
| 자동매매 시스템 구성에 **파이썬 코딩 + 금융 지표 분석 능력** 요구 | 일반인 진입장벽 ↑ |
| **비선형 시장 변동성**에 고정 수식만으로는 대응 불가 | 전략 성과 저하 |
| 상용 플랫폼의 **폐쇄적 구조** | 보안 리스크 / 거래소 종속 |

### 2.3 해결안

**3-Pillar 통합 솔루션**:

1. **GPT 기반 자연어 전략 추천 + No-Code 편집기** → 진입장벽 해소
2. **강화학습 엔진** → 시장 변동성 능동 대응
3. **비수탁형 멀티 API 연동 구조** → 보안 + 자유도 확보

### 5.5 설계 원칙

1. **3-Click Rule** — 핵심 기능은 홈에서 3클릭 이내
2. **빈도순 배치** — 사이드바 상단부터 매일 쓰는 메뉴
3. **위험 격리** — API 키·결제는 2FA 재인증 후 접근
4. **실패 가시성** — 예외 화면은 단순 에러가 아닌 자동 대응 과정 노출

## 9. 핵심 의사결정 근거

핵심 의사결정과 그 근거를 기록한 **Decision Log**입니다. 향후 재검토 시 출발점이 됩니다.

### 🎯 D-01. 왜 비수탁형(Non-custodial) 구조인가?

**결정:** 자산은 사용자의 거래소 계정에 그대로 두고, API 키만 받아 매매 신호를 전달한다.

**근거:**
- 박서연 페르소나의 핵심 Pain Point ("자산을 앱에 맡기는 게 찝찝함")
- 기존 상용 플랫폼 대비 명확한 차별화 포인트
- 가상자산이용자보호법 규제 부담 ↓

**트레이드오프:** 사용자가 API 권한 관리에 신경 써야 함 → 가이드 + 권한 검증 UX로 완화

### 🎯 D-02. 왜 GPT 자연어 + No-Code 블록 편집기 듀얼 인터페이스인가?

**결정:** GPT 채팅과 블록 편집기를 좌우 분할로 동시에 노출한다.

**근거:**
- 초보자(GPT 추천만 사용) ↔ 중급자(블록 미세조정)를 동시에 만족
- 양방향 동기화 → 채팅이 곧 살아있는 문서
- 단일 인터페이스 대비 학습 곡선 완만

**대안:** 단계별 마법사(Wizard) 형태도 검토했으나, 박서연의 "빠르게 만들고 싶음" 니즈와 맞지 않아 기각

### 🎯 D-03. 왜 강화학습 엔진을 별도 명세화했는가? (F-07-03)

**결정:** Mission 02 CUT 5에서 도출한 "RL 엔진 자동 변동성 대응"을 정식 기능 명세로 격상.

**근거:**
- 단순 자동 재연결을 넘어 "지능적 적응"을 보여주는 차별점
- 와이어프레임 W6에서 사용자에게 학습 과정을 가시화 → 신뢰 형성

**관련 산출물:** Mission 02 CUT 5 → Mission 03 노드 5.3 → Mission 04 Flow 2 → Mission 06 W6

### 🎯 D-04. 왜 사이드바 메뉴 순서를 사용 빈도로 정렬했는가?

**결정:** `대시보드 → 전략 → 거래 → 거래소 → 설정 → 지원` 순.

**근거:**
- 매일 들어가는 메뉴(대시보드/전략/거래)를 위쪽 배치
- 1회성 진입(거래소 연동)은 중간
- 가장 덜 쓰는 지원/설정은 하단

**관련 산출물:** Mission 03 그룹핑 논리표 → Mission 06 W2/W6 사이드바

### 🎯 D-05. 왜 예외 화면(E.3)을 일반 에러 페이지가 아닌 "복구 진행 트래커"로 설계했는가?

**결정:** 4-step 진행 인디케이터로 시스템이 무엇을 하고 있는지 보여준다.

**근거:**
- 박서연이 회의 중에 알림을 받았을 때 **불안하지 않게** 만드는 것이 핵심 UX
- 단순 "에러 발생" 메시지는 사용자의 자동매매 신뢰를 깨뜨림
- "사용자 개입 불필요" 카피로 안심시키되, 상세는 RL 모니터로 유도

**관련 산출물:** Mission 02 CUT 5의 "푸시 알림으로 사후 통보" 설계 의도 → Mission 06 W6 정밀화

### 🎯 D-06. 왜 첫 전략 배포에 2FA를 강제하는가?

**결정:** Flow 1의 마지막 단계 "전략 배포(실거래 ON)"에서 2FA 재인증.

**근거:**
- 실거래 = 실제 자산이 움직이는 시점, 비가역적 액션
- 위험 격리 원칙 적용 (Mission 03 설계 원칙 #3)
- 보안 사고 시 책임 소재 명확화
